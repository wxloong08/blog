## 先把话说在前面

这不是一篇“证明 url-reader 很强”的文章。  
相反，我更想把边界说清楚：**现在的数据样本非常小，只够说明当前方法怎么测、第一轮跑出来是什么结果，还远远不够下大结论。**

目前只测了 3 个 URL，而且都是我自己在 Claude Code 调研时真实访问过的页面。  
它更接近“第一轮工作日志”，不是完整 benchmark 报告。

## 我到底想测什么

对我来说，一个网页读取工具值不值得继续维护，主要看三件事：

1. **能不能成功拿到可用内容**
   - 不是反爬页面
   - 不是空白
   - 不是只抓到一层包装壳

2. **最后是哪个策略真正完成了任务**
   - 是 `Firecrawl`
   - 还是 `OpenCLI Browser`
   - 还是 `Jina`
   - 还是 `Playwright`

3. **清洗之后到底减少了多少无用字符**
   - 这不是严格 token 统计
   - 但至少能看出“正文以外的壳子”被剥掉了多少

所以这轮 benchmark 记录的是：

```text
URL
-> 平台识别
-> 实际命中策略
-> 总耗时
-> 清洗前字符数
-> 清洗后字符数
-> 字符压缩比例
```

## 第一轮样本

| 平台 | URL | 类型 |
|---|---|---|
| 知乎问答 | `/question/10434775822` | 问答 |
| Reddit | `/r/ClaudeAI/comments/1qfosa6/` | 论坛 |
| SSDNodes 博客 | `/blog/claude-code-pricing-in-2026...` | 技术博客 |

这 3 个 URL 不是为了 benchmark 专门挑的，而是我在实际调研中确实访问过的页面。  
所以它们的意义不是“覆盖面广”，而是“更接近真实使用时会遇到的问题”。

## 测试环境

本轮环境如下：

```text
Firecrawl API Key: 已配置
OpenCLI: 1.7.7
Jina Reader: r.jina.ai
Playwright: 已安装，但本轮未命中
操作系统: Windows 11
网络: 国内家庭宽带 + 代理
```

这里要特别说明两点：

- 这个结果会明显受网络和代理质量影响
- 它也会受平台当天的风控状态影响

所以你不应该把下面的数字理解成“稳定均值”，而应该理解成“在 2026-05-07 这次环境下跑出来的一次真实结果”。

## 实际运行方式

这轮数据直接来自项目里的 benchmark 脚本：

```bash
python scripts/benchmark.py
```

结果会落到：

```text
benchmark_results.json
```

这一点我觉得很重要，因为至少别人以后能按同样方法重跑，而不是只能看一张表。

## 第一轮结果

### 原始结果

| URL | 平台 | 命中策略 | 耗时 | 字符压缩 |
|---|---|---|---|---|
| 知乎 `/question/10434775822` | 知乎 | OpenCLI Browser | 27.1s | 50% |
| Reddit `/r/ClaudeAI/comments/1qfosa6/` | Reddit | OpenCLI Browser | 14.7s | 1% |
| SSDNodes 技术博客 | 通用 | Firecrawl | 1.2s | 7% |

### 这组数先说明什么

先不下大结论，只看这 3 条结果本身：

- **知乎这条最能说明清洗的价值**
  - 原始页面里有很多壳子
  - 最后只保留了问题和已加载回答
  - 所以压缩比能到 `50%`

- **Reddit 这条说明“成功抓到”和“清洗得足够好”是两回事**
  - 它是成功的
  - 也确实命中了 `OpenCLI Browser`
  - 但压缩率只有 `1%`
  - 这基本等于在提醒我：`Reddit` 现在还缺专门提取器

- **SSDNodes 这条更像 Firecrawl 的典型优势样本**
  - 快
  - 原始返回已经比较干净
  - 所以后处理空间很小

## 这一轮能得出的谨慎判断

我觉得这一轮只够支撑 3 个很保守的判断。

### 1. OpenCLI Browser 目前是有实际价值的

至少在这轮样本里，`知乎` 和 `Reddit` 都是靠 `OpenCLI Browser` 成功拿到内容的。  
这说明它不是摆设，而是真的填补了 `Jina` 失效之后的那一层空白。

### 2. “成功率”不能替代“内容质量”

`Reddit` 是个很好的例子。  
它 technically 成功了，但压缩率只有 `1%`，说明“页面是读到了”，不代表“正文已经被整理好了”。

### 3. Firecrawl 对普通公开技术博客非常省心

在这条 SSDNodes 样本上，`Firecrawl` 的表现就是：

- 速度快
- 返回干净
- 后处理几乎只做收尾

但这也只是针对这类站点，不应该直接外推到所有站。

## 这一轮不能得出的结论

这部分我反而觉得更重要。

基于这 3 个 URL，我**不能**认真地说：

- “中文站成功率已经很高”
- “Jina 对中文和 Reddit 基本不可用”
- “Firecrawl 是最优方案”
- “OpenCLI 比 Playwright 更值得优先”

这些判断都需要更大的样本、更稳定的网络条件和重复测试才能说。

现在最多只能说：

> 在这轮 3 个真实样本里，`OpenCLI Browser` 解决了知乎和 Reddit 的读取问题，而 `Firecrawl` 在一篇普通技术博客上表现最好。

## 这轮 benchmark 反而暴露了什么

这次跑完以后，我更清楚接下来该补什么了。

### 1. Reddit 还需要专门清洗

压缩率 `1%` 已经说明问题了：  
读取成功不等于现在的 Reddit 输出已经足够干净。

### 2. 样本量太小

3 个 URL 只能验证“链路通不通”，不足以验证“整体策略设计是不是合理”。

### 3. 需要重复测试

尤其是依赖浏览器渲染和代理网络的策略，单次耗时波动会很大。  
后面至少应该同一 URL 连跑 3 次再取平均。

## 下一步怎么补

下一轮我会优先补这些：

1. 把样本扩到 `10+` 个 URL
2. 加上 `V2EX`、`NodeSeek`、`HostLoc`、`微信公众号`
3. 同一 URL 重复跑 `3` 次
4. 区分“读取成功率”和“清洗质量”
5. 单独补一轮 `Reddit` 评论提取器再测一次

## 小结

这篇最有价值的地方，不是那张结果表，而是我现在已经知道：

- 哪些结论还不能说
- 哪些样本最值得补
- 哪些策略问题是“抓不到”
- 哪些是“抓到了但还不够干净”

对 `url-reader` 这种工具来说，我现在更相信后一种指标。  
因为它最终服务的不是“网页有没有抓下来”，而是“能不能给 agent 一份更可用的阅读输入”。

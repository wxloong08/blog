> 我从零写了一个通用的 AI Agent CLI，叫 loongcli，架构上参考了 Claude Code，但面向 DeepSeek 生态、主跑 deepseek-v4-pro。我每天用它开发它自己。这篇讲的是我在它身上发现的、迄今最严重的一个 bug——之所以严重，不是因为它难修，而是因为它**完全不报错**。

## 一、症状：Agent 开始「精神分裂」

最初的线索不在 loongcli，而是在 Claude Code 身上。

我用 Claude Code 跑一个多步骤的工作流技能（一个十几步、带状态机的长流程）。跑到中途，对话很长，触发了一次上下文压缩（compact）——把前面几十轮对话摘要成一段，腾出窗口。压缩之后，我注意到 Agent 开始「精神分裂」：它对同一件事，压缩前一个说法，压缩后另一个说法，像是忘了自己几分钟前定下的规矩。

这其实是可以理解的。compact 是有损压缩，几十轮对话被摘要成几百字，技能里某个具体步骤的细节，被摘没了。Agent 拿着一份梗概在继续干活，自信地偏离了原始指令。

我没法改 Claude Code，但我有一个自己的 Agent。于是我回头问：**loongcli 有没有同样的毛病？**

答案是：不但有，而且更糟。

## 二、第一层根因：上下文忘了

loongcli 的 compact 逻辑和直觉一致：当 prompt token 超过阈值，调一次 LLM 把历史摘要成一段结构化总结，然后用「系统提示 + 摘要 + 最近 3 轮对话」替换掉整个消息列表。被摘要掉的中间过程，从 Agent 的工作上下文里消失了。

技能指令也是这中间过程的一部分。技能内容是通过一次工具调用加载进上下文的，compact 时它和其它历史一起被摘要——摘要器最多在「当前进度」里留个梗概，技能那几千字的具体规则，没了。

到这里，问题和 Claude Code 看起来是同一个：**上下文有损压缩导致契约细节丢失**。这已经够烦了。但当我顺着代码往下看「压缩之后这些历史去哪了」，发现了第二层，那才是真正吓人的地方。

## 三、第二层根因：磁盘也忘了

loongcli 每跑完一轮，会把完整对话持久化到会话文件，用于之后的 `--resume`。这个持久化函数很朴素：

```python
def _persist(self):
    if self.conversation_store:
        self.conversation_store.save(self.messages)
```

它保存的是 `self.messages`——也就是 Agent 当前的工作上下文。

而 auto-compact 干的事是**就地替换** `self.messages`：把它换成压缩后的摘要版本。

把这两件事连起来：compact 发生后，`self.messages` 已经是压缩版了；下一次 `_persist()`，就把这个压缩版**覆写**进了会话文件。

也就是说——**被压缩掉的完整历史，不只是从 Agent 的上下文里消失，它在磁盘上也被压缩版覆盖了。** 原始对话，物理删除，不可恢复。`/clear` 命令也有同样的问题。

最讽刺的是，我那阵子刚给 loongcli 做了一个 Web UI，主打功能就是「浏览完整会话历史」。在这个 bug 面前，那个功能是个笑话：任何被压缩过的会话，磁盘上只剩摘要，所谓「完整历史」根本不存在。

这就是我说它严重的原因。它不抛异常、不打日志、不弹警告。你以为历史好端端躺在硬盘上，其实每一次自动压缩都在悄悄销毁证据。**静默的数据损坏，比会崩溃的 bug 危险得多。**

## 四、退一步：压缩到底该「丢」什么？

定位到根因，最容易的反应是「那就在 compact 前备份一下原始消息」。这能修第二层。但我强迫自己先退一步，想清楚一个更本质的问题：

**压缩必然有损。问题从来不是「要不要丢」，而是「丢什么、丢了能不能找回」。**

对话历史不是均质的数据流。我把它按「丢失后果」分成三类：

- **契约类**——技能指令、用户立下的约束和决策。这类内容低熵、承重，丢了的后果是**静默腐败**：Agent 行为悄悄漂移，不报错。这正是开头那个「精神分裂」。
- **事实类**——工具输出、读过的文件内容、报错原文。这类体积大，但大多**可以重新获得**：文件可以再读一遍，命令可以再跑一次。丢了不致命，但要能找回。
- **叙事类**——中间的推理过程、试错。摘要就是它的正确归宿，丢掉细节、留下脉络，没问题。

这个分类是后面所有设计的地基。它说明：**「备份原始消息」只解决了一半**。备份让历史在磁盘上可追溯（事实类、叙事类有救了），但它救不了契约类——因为契约类的危险不在「存档丢了」，而在「Agent 的当前上下文里没有了」。Agent 不会去翻存档，它根本不知道自己忘了。

## 五、三层修复

于是修复是分层的，每一层对应一类内容。

**第一层，历史保真（救事实类 + 叙事类）。** 在所有会「就地改写 messages」的地方——auto-compact、手动 compact、clear——动手之前，先把原始消息追加进会话文件的一个归档区。持久化函数从「整体覆写」改成「读-改-写」，保证 `messages` 字段只代表当前工作上下文，而「完整历史 = 所有归档段 + 当前上下文」。从此磁盘上的历史不可变、只增不减。

**第二层，契约保活（救契约类）。** 这一层是治本。压缩之后，从磁盘**重新读取**当前活跃技能的原文，重新挂载回上下文，并明确标注「以此原文为准，摘要里的梗概仅供参考」。同时把摘要模板改成「只记录执行进度，不要复述技能指令」——指令交给重注入，摘要只管进度，两者分工不打架。

**第三层，按需检索（补事实类的找回）。** 加一个 `search_history` 工具，让 Agent 能在完整历史（含归档段）里做关键词检索。当它需要回忆某个早前的具体细节——那个报错的原文、用户当时的原话——而摘要里没有时，它可以主动去捞。

## 六、最关键的判断：为什么契约用「重注入」而不是「检索」？

这是整个设计里我最想讲的一点，因为它区分了「想清楚了」和「凭感觉堆功能」。

既然加了 `search_history`，为什么不让 Agent 也用它去捞回技能指令，省掉第二层的重注入？

因为**检索的前提是「你知道自己缺什么」**。

- 「刚才那个报错的原文是什么」——这是个 *known unknown*：Agent 知道有个报错、知道自己想不起细节了，于是它会去检索。检索能救。
- 但技能指令丢失是 *unknown unknown*：Agent **不知道自己忘了**第七步的那条规则。它不会去查，因为它压根没意识到有东西需要查——它只会自信地、错误地继续往下做。

对 unknown unknowns，事后检索永远追不上事前保活。所以契约类必须**结构性地塞回上下文**（不管 Agent 要不要），而事实类才交给**按需检索**（Agent 想起来就去捞）。两个机制不是二选一，是各管一类。

这个「unknown unknown 用重注入、known unknown 用检索」的分工，是这次修复真正的认知产出。它比任何一行代码都重要。

## 七、诚实的结尾：我怎么验证它有用？

到这里，故事可以漂亮地收尾了：发现 bug、三层修复、漂亮的设计哲学。但如果我就这么结束，那是在骗面试官，也是在骗自己。真实的结尾没那么漂亮，而它恰恰是这篇文章里我最想让你看到的部分。

修完之后，我给第二层（契约重注入）写了一个压力测试，想用实验**证明**它有用：造一个合成技能，规则只写在技能文件里——创建的每个文件第一行必须是某个验证码（这个验证码不出现在任何任务指令里）。然后把压缩阈值调到很低，跑多轮强制触发多次压缩，看压缩之后 Agent 创建的文件还带不带验证码。再设一个对照组：**关掉重注入**，看会不会违约。

预期是：对照组该翻车，实验组该全过——干净的 A/B，证明重注入是那根救命稻草。

结果：**两组都全过。** 对照组没翻车。

也就是说，我的测试**没能证明重注入是必要的**。

我去查为什么。根因是：在我的 Agent 里，技能加载产生的工具结果不在「可回收清理」的名单里（它被当成不可重取的内容保护着），所以它会**完整地进入摘要器的输入**。而我用的验证码是一条非常显眼的硬规则（「忘记 = 任务失败」），摘要器自然把它保留了下来。换句话说——**对「显眼的硬规则」型契约，摘要这一层自己就够了，根本轮不到重注入兜底。**

事情还没完。我在复盘这个结果时，又揪出一个更尴尬的 bug：重注入的附件用的标题格式，和我的「活跃技能检测器」认的格式**对不上**。后果是——第一次压缩后，原始技能工具结果被摘要掉了，第二次压缩时检测器找不到活跃技能，**重注入就静默停了**。也就是说，在那个跑了 5 次压缩的实验里，第 2 到第 5 次，重注入**根本没在运行**。我精心设计的「治本之策」，因为一个标题格式不匹配，悄悄罢工了，而所有测试还是绿的——又一个静默失效，和我一开始排查的那个 bug 同一个性质。

我把它修了（标题改成检测器认得的规范格式，并加了一条回归测试：断言检测器能从重注入文本自身识别出技能名，防止以后再改格式时复发）。

但更重要的是，我没有把第二层粉饰成「已验证的治本之策」。诚实的结论是：

- **第一层（历史保真）是已证实的真实 bug 修复**——磁盘覆写确实发生，归档后历史可还原，有测试和 UI 实测。
- **第三层（检索）是有单测覆盖的新能力。**
- **第二层（契约重注入）目前是「未被证伪」，不是「已被证明」。** 在当前的压缩参数下，它是一层面向未来的防御——一旦技能内容被纳入清理、或技能原文超出摘要的保留能力、或经过多代压缩累积衰减，它才会真正兜底。我的压力测试用的是「显眼硬规则」型契约，测不出它的价值；要测出来，得用「细节型契约」（藏在长技能里某个不起眼的具体步骤，摘要会丢的那种）。这是下一步。

我把那个压力测试保留了下来，但重新定位为「压缩后契约 + 归档保真」的回归测试，而不是「重注入必要性」的证明。它锁定的不变量是真实的：压缩后契约仍在、且归档段数等于压缩次数。

## 八、可迁移的认知

抛开 loongcli 这个具体项目，这次排查留下三条我会带走的认知：

1. **静默失效是最贵的 bug。** 这个故事里出现了两次：compact 覆写磁盘、重注入因格式不匹配罢工。两次都不报错、测试都是绿的。会崩的 bug 会自己喊救命，静默的不会——所以关键不变量要有断言去钉死它（归档段数 == 压缩次数、检测器认得重注入文本），让静默失效变成会响的红灯。

2. **想清楚「丢了能不能找回」，比想「要不要丢」更重要。** 有损系统的设计核心是分类：哪些是 unknown unknown（必须结构性保活）、哪些是 known unknown（可以交给检索）。把这两类混为一谈，要么过度保留浪费资源，要么该保活的丢了还以为有检索兜底。

3. **诚实区分「未证伪」和「已证明」。** 我可以把三层修复都写成「治本」，没人会去查我的压力测试。但一个连自己都骗的工程师，做不出可靠的系统。我宁愿告诉你「第二层我没能证明它必要，而且它一开始还罢工了」——这比一个漂亮但经不起追问的结论，值钱得多。

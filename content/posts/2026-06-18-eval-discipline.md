> loongcli 开发笔记第四篇。前三篇讲的都是单点 bug，这篇讲一次完整的评测：我想给自己的 Agent 找弱项，过程中两次差点下错结论，一次被自己的数据推翻、一次被一句质疑点醒。给未来的自己留一句话——**判断让位于测量，而且要敢被自己的数据打脸。**

## 起点：一个悖论

我想让 loongcli「更准地完成任务」。第一步不是改代码，是**找到它哪里不准**——给它造更强的测试。

但一上来就撞到悖论：我每天用它开发它自己，**没发现什么痛点**。那"更准"从何谈起？我的判断是：不是没弱项，是日常使用没把它压到边界。所以要主动造能压到边界的测试。

我已有一个自建的 SWE-bench 式评测框架（从 git 历史挖 fix commit 生成 fail-to-pass 任务），但它**饱和了**：DeepSeek 几乎满分。一把所有人都得 100 分的尺子，测不出任何东西。

## 第一刀：模糊 prompt —— 假设被否

我看现有任务，发现它们的 prompt 把解法都写出来了（"把每行限制在 150 字符内"——这是答案不是问题）。假设：**难度太低是因为 prompt 泄露了解法**。

于是把一个 prompt 从"给解法"改成"只给现象"，跑对照。结果：**没用**，19 步 vs 21 步，都解决了。

原因我应该早想到：fail-to-pass 范式下 agent 能读测试，**测试即规格**——prompt 给不给解法，它从测试文件就反推出要什么。第一个假设，否。

## 第二刀 + 翻车（一）：我把噪声当成了弱项

第二个假设：真正考验 Agent 的是**上下文压缩**（compact）——这是 DeepSeek 模型帮不上、纯靠 harness 的能力。我把压缩阈值从一百万调到一万五，让任务频繁触发压缩，跑了一个任务：

步数从 19 涨到 **35（+84%）**。我兴奋地记下"找到了 compact 的效率弱项"。

然后我多跑了几个任务确认——**全翻了**。4 个任务在压缩压力下步数不增反降，同一个任务同样配置，步数一次 19、一次 23，压缩组一次 35、一次 21。**方差近一倍。那个 +84% 纯粹是噪声。**

这是这篇里我最想记下的一次打脸：我在**前三篇 blog 里反复写"判断让位于测量、n=1 不能下结论"，结果自己又踩了**。我虽然标注了"n=1 待确认"，但结论的语气已经倒向"找到弱项"。教训补一刀：**标注不够，结论的语气也要匹配证据强度。** 要不是多跑那一批，我就要带着一个噪声结论，去"优化"一个根本不存在的问题了。

## 翻车（二）：我以为在测满载，其实在测频率

正当我准备收尾，我停下来问了自己一个问题：**这个测的，和真实使用一样吗？**

不一样。我把压缩阈值压到一万五，但窗口还是一百万——等于在**窗口 1.4% 的地方**就触发压缩，每次只压一点点历史。而真实的"满载压缩"是把**接近一百万 token 的历史压成几千字摘要**，信息损失率天差地别。我测的是压缩**频率**，根本没碰到真正的难点：压缩**规模**。

我以为在测 A，其实在测 B。这是造测试最隐蔽的坑。

## 一个反直觉的发现

修正测法时，我想用"读大文件"把上下文撑到接近满窗口。结果读了 6 个 9000 字符的大文件，上下文峰值**才 16K**——撑不上去。

根因：loongcli 对工具结果做**入口截断**——单条结果超过 8000 字符就只留 2000 字符预览（附一句"需要完整内容可重新调用工具"），单轮所有工具结果还有 3 万字符的总上限。所以读进来的大文件根本没完整进上下文，6 个 9000 字符的文件各被截成 2000 字符预览。要撑大上下文，得用**不会被截断的内容**（对话里的 user/assistant 消息）。

这反过来是个有价值的结论：**至少在以工具调用为主的编码任务里，loongcli 很难「满载」**——因为 token 的大头（读文件、命令输出）在入口就被截断，上下文涨不到满窗口。我担心的"满载大压缩"，在这种场景下恰恰被这个设计规避掉了大半。（纯靠多轮对话堆积当然还是能撑满的——所以我后面改用对话内容来撑。）

## 顺手回答了一个老问题：compact 后还记得开头吗？

换成用对话内容撑上下文，我撑到 51K、触发了 10 次连续压缩，测一个具体问题：**第一轮埋的一个代号，经过 10 次压缩后，Agent 还访问得到吗？**

结果：**5/5 可达**，而且纯靠摘要被动保留——根本没动用「检索归档」的兜底工具。对"开头明确交代的信息"，loongcli 保留得很好。

但要诚实标边界：代号在第一轮消息**开头、很显眼**，摘要倾向保留这种；埋在长文本中间的"细节型"信息没测，可能会丢。而且真正的残余风险是 **unknown-unknown**——如果摘要真丢了某个细节、而 Agent 不知道自己忘了，它就不会去检索。这是所有 Agent 的共同难题，不是 loongcli 特有。

## 最后一课：46% 的缓存命中率吓了我一跳

当天的 API 用量面板显示缓存命中率只有 **46%**，远低于我平时测的 90%+。第一反应是"缓存优化失效了？"

不是。那是**当天所有测试的总和**——大量独立短会话冷启动、探针每轮灌新内容、压缩探针故意破坏前缀，全是缓存的最坏情况。真实使用（单个长会话）命中率依然 90%+。

**测试场景的资源特征（缓存、上下文、成本），不等于真实使用。** 别用评测数据去判断产品的真实表现。

## 可迁移的认知

1. **判断让位于测量，且要敢被自己的数据推翻。** 这次两次：n=1 的"弱项"是噪声，自以为的"满载测试"测的是频率。能被推翻，是因为我每次都留了"再测一次"的余地。一个把判断焊死的工程师做不出可靠系统——哪怕那个判断是他自己写在 blog 里的。

2. **造测试比想象的难，因为你常常在测 B 却以为在测 A。** 模糊 prompt 测的还是模型、低阈值测的是频率不是规模、读文件测不出满载。设计任何测试前，先逼问一句:**我到底在测什么?**

3. **测试场景 ≠ 真实使用。** 评测时的缓存命中、上下文规模、token 成本都是测试特有的。判断产品要用真实数据。

4. **有时"没找到弱项"就是结论。** 我没找到 loongcli 的稳定弱项——它在能造出的任务上又准又稳。这不是失败，是个诚实的结论:这个工具已经够好了，与其为它硬找毛病，不如把力气花在别处。

---

*这次评测花了我大约八块钱的 API 费用，跑了几十次任务和探针。两个新探针（compact 压力注入、长程记忆可达性）都留在了仓库里。最值钱的产出不是某个数字，是又一次被提醒：最难纠正的，是你自己深信不疑的判断。*

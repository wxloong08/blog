> 这是 loongcli 开发笔记的第二篇(上一篇讲 compact 摧毁历史)。记给未来的自己:一个"凭第一性分析觉得不值得做"的优化,怎么因为我没固执己见、花了几毛钱测了一下,变成了"该做、做对了"。核心教训不是那个优化本身,是**判断该让位于便宜的测量**。

## 起点:一个跑题跑出来的问题

本来在聊一个无关的事(要不要给 agent 加"自动等待后台任务"的能力,我判断不必要)。聊着聊着拐到:loongcli 的固定 prompt 占用大概 57K token,如果技能变多、记忆变多,这个固定占用会不会膨胀?

我查了一下组装逻辑,答案是:**不会失控**——记忆索引有 25K 字节硬上限,技能列表是懒加载、只列名字。但更重要的是,我意识到自己差点回答错重点:

**固定占用涨几 K 字节根本不是成本大头。** DeepSeek 的前缀缓存命中价是 ¥0.025/M、未命中 ¥3/M,差 120 倍。只要固定前缀稳定,它几乎免费。真正烧钱的不是"占用变大",是"**前缀不稳定导致缓存失效**"。

顺着这个往下想,我盯上了一个一直没注意的东西:记忆召回(recall)。

## 第一个判断:position 1 注入是缓存反模式

loongcli 每轮用户输入会自动召回 top-5 相关记忆,注入上下文。我去看注入位置——它被插在 **position 1**:主 system prompt 之后、**整个对话历史之前**。

这里要先讲清楚前缀缓存的机制(它决定了一切):缓存是**前缀连续匹配**的。从头逐 token 比对,匹配的前缀复用 KV,**第一个不一样的 token 往后,全部失效重算**。

所以把"每轮都可能变"的召回内容插在 position 1,意味着:只要这轮召回和上轮不同,**从 position 1 往后的所有东西**(整个对话历史 + 工具结果)的缓存全部作废。这在机制上显然是反模式——你把最不稳定的东西放在了几乎最前面。

对应的原则也很清楚:**稳定的放最前,每轮变的放最后。** 变化内容越靠末尾,它失效的范围越小。

## 该不该改?我的第一反应是"不值得"

机制上是反模式,但要不要改,我的第一反应是**先别动**,理由有两条:

1. **收益是未知的经验量。** position 1 注入不是"每轮都失效"——只有"这轮召回和上轮内容不同"时才失效。而召回在很多情况下是稳定的(同一个任务里,相关记忆往往是同一批)。所以真实损失 = 召回的逐轮变化率 × 历史大小,这是个我**不知道、推不出来**的经验数。
2. **绝对成本可能很小。** DeepSeek 缓存便宜,即使失效,几分钱一轮。对一个个人工具不是痛点。

但就在写下"不值得改"的瞬间,我意识到一件事——**这是个经验问题,不是推理问题**。我在用直觉给一个"必须测才能知道"的量拍板。这恰恰是第一性原则最忌讳的:在经验问题上假装推理能解决。

而测量几乎免费:`CostTracker` 本来就记着 `prompt_cache_hit_tokens`。所以正确的动作不是"改"或"不改",是**先花几毛钱把收益测出来**。

## 测量:数据部分推翻了我

我写了个 A/B 探针(`bench/cache_probe.py`),不碰产品代码,在脚本里直接构造两种布局:

- 布局 A(现状):`[system, recall(每轮变), ...历史..., user]`
- 布局 B(优化):`[system, ...历史..., recall(每轮变), user]`

唯一变量是召回位置。用真实 system prompt、真实召回格式,连续发 10 轮(对话累积),读 DeepSeek 返回的真实缓存命中。为了逼出位置影响的上界,我故意让每轮召回不同的记忆。

结果:

| | 第 2 轮起平均命中率 | 10 轮 input 成本 |
|---|---|---|
| A(position 1) | **53%** | ¥0.0676 |
| B(末尾) | **99%** | ¥0.0028 |

**末尾注入省 96% 的 input 成本,命中率差 46 个百分点。**

这部分推翻了我的"不值得改":

- "收益是经验量"——对,但测出来这个量很大(46pp),而且我还发现 A 的 53% 是被我的实验**高估**的:实验里召回每 5 轮规律循环,A 靠这种复用才到 53%;真实场景召回不规律,A 只会更接近"只命中 system"的低位。
- "绝对成本小"——半对。这个实验 prompt 才 3–6K token,所以省的是半分钱/轮,确实小。但 46% 是**比例**,会随 prompt 规模线性放大。loongcli 有 compact 机制专门服务长会话——一旦会话长到几万、十几万 token,46% 的命中差就是每轮多算几万 token + 明显的 TTFT 变长。**而长会话恰恰是这个机制存在的理由。** 我之前说"成本小"只在小对话成立,漏了规模依赖。

关键洞察落在这:**缓存维度上,末尾注入是帕累托改进**——召回稳定时两种都命中(持平),召回变化时末尾远好于 position 1。它没有下行风险,只有上行收益,且会话越长收益越大。

判断于是从"不值得改"翻转成"该改"。不是因为我改了主意,是因为**数据说话了**。

## 改之前的最后一道坎:注意力,单元测试管不了

把召回从 position 1 挪到末尾,代码改动很小。改完单元测试全过(1138 个),缓存收益实测确凿。但我没急着提交,因为还有一个维度单元测试**根本碰不到**:

**注意力。** 召回从"紧贴 system prompt 的靠前位置"挪到"对话末尾",模型对这些召回记忆的**重视程度**会不会变?单元测试只能证明"召回注入还在、不重复",证明不了"模型还在用它"。

这就是这篇笔记里我最想记下来的方法——**怎么测一个不可观测的东西**。

模型内部的 attention weights 不可观测(闭源,且就算拿到也难解释)。所以不能直接测"注意力"。我把它换成一个**行为代理**:如果模型"重视"召回的记忆,它就该能用上记忆里的信息答对问题;答不对=没用上=没重视。**把不可观测的"注意力",转成可观测的"答对率"。**

然后三个设计保证这个代理有区分度(吸取上一个教训:compact 压测那次任务太简单,两边都满分,什么都没证明):

1. **关键事实人造、不可猜**——灰度观察窗口 6 小时、回滚阈值 0.3%。如果用模型能从常识猜到的事实,它不靠召回也答得对,就测不出注入位置的影响。
2. **埋在 5 条召回的中间 + 4 条干扰**——模拟真实 top-5,测的是"从一堆里提取关键那条"。
3. **加几轮无关对话历史**——这是关键。注入位置的差异(历史前 vs 历史后)只有存在历史时才体现;历史把"召回离当前问题的距离"拉开。没历史两种位置几乎一样,测了白测。

同问题、同记忆、同历史,唯一变量是位置,各跑 6 次取答对率。判读规则:末尾 ≥ position 1 就算不退化;末尾明显更低才是退化。

结果:

| | 答对率 |
|---|---|
| position 1(旧) | 5/6 |
| 末尾(新) | **6/6** |

末尾不差,甚至略好。**诚实标注边界**:n=6 是小样本,position 1 那次失败其实是个空回答(更像 API 偶发,不是系统性退化),所以我不把"末尾更好"当强结论——准确说法是"末尾至少不差、无退化迹象"。但这已经足够:结合缓存维度的确凿收益,改动安全。提交。

## 可迁移的认知

1. **经验问题别拍脑袋,而度量往往很便宜。** 我差点用直觉(还带着像样的第一性分析)给一个"必须测才知道"的量下结论。区分"推理能定的"和"必须测的",是这条线最重要的纪律。这次测量总共花了几毛钱。

2. **判断被数据推翻是好事,不是打脸。** 我两次判断被实测修正过(这次的"不值得改",还有上一篇 compact 里"重注入有用")。能被推翻,是因为我没把判断焊死,而是留了"先测"这一步。一个不肯被数据修正的工程师,做不出可靠的系统。

3. **前缀缓存 = 稳定的放前面,变化的放后面。** 任何每轮变化的动态注入(召回、动态计划、临时上下文),放得越靠末尾,缓存失效范围越小。这是个能反复用的结构原则。

4. **不可观测的量,用它的行为后果做代理。** 测不了"注意力",就测"能不能用上召回信息答对题"。把抽象的、内部的量,锚定到一个具体的、可判分的外部行为上——这是给"玄学维度"做实验的通用套路。

---

*两个探针(`cache_probe.py` 量缓存命中、`recall_attention_probe.py` 量召回利用)留作了 loongcli 的可观测工具,在仓库里。以后任何动 prompt 结构的改动,都能用它们量化影响,而不是凭感觉。*

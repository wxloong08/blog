> loongcli 开发笔记第五篇。这篇的主角不是某个 bug，是「防御本身的盲区」——以及我在追查它的路上，又几次「以为自己看全了」。给未来的自己：**最危险的不是忘了设防，是设了防、却以为防全了。**

## 起点：一个我亲手设防的地方，还是漏了

我用 Claude Code 跑一个找工作的工作流 skill（jobhunter）：自动监控招聘消息、筛掉不合适的、对值得的做背景调查。skill 里有一份黑名单——我亲身踩过坑的公司，规则是一律跳过、不分析。

某天它从一次上下文压缩（compact）恢复后，对一家公司做了完整、专业的分析。问题是这家公司**就在黑名单上**。我点出来，它承认：压缩恢复后没重新加载黑名单。更糟的是——**这是一周内第二次**，第一次的教训我还专门记进了文档。

## 我以为我防过了——这恰恰是问题所在

关键在于：我**早就给这个 skill 写过一节「上下文恢复协议」**，专门防 compact 丢状态。它白纸黑字写着"压缩后强制执行：读进度文件、读决策文件、恢复意图"。

我设了防。可 bug 还是发生了，两次。

## 插一段：我想验证我自己的 Agent 会不会也犯

loongcli 是我从零写的 Agent。我搭测试想复现这个 bug。第一次又栽在测试设计上——我让合成 skill "每次都重读黑名单"，那当然不会忘；而真实的失效模式是"开头加载一次、之后凭记忆"。**我以为在测 A，其实在测 B**——这个坑我上一篇 blog 刚写完，转头又踩了。

改对、加到最难（把黑名单藏进 80 项清单的中间），loongcli 几个变体都不犯。我一度得出结论：**loongcli 架构更强，结构性地避开了这个 bug。**

## 然后我去读了那个 skill 的源码，发现我又错了

真相不在 harness（无论 Claude Code 还是 loongcli），在**那节恢复协议本身**：

它列清了"压缩后要恢复什么"——扫描进度、用户决策、未持久化的表态。**唯独漏了「重读黑名单」。**

Claude Code 没有不守规矩，它**忠实执行了我写的协议**：恢复了进度、恢复了决策。但黑名单不在清单上，于是它"按协议恢复完毕、以为齐了"，而黑名单还是空的。

**我设的防御，自己漏了一项。** 而且正因为我"设过防"，我反而更难想到去查它——直到它坑了我两次。

我那个"loongcli 架构更强"的结论也跟着塌了：loongcli 跑同一个 skill，会执行同一个有盲区的协议，一样会漏。它在我的测试里没犯，只是因为"摘要碰巧保留了原文"这条侧路救了场，**不是协议对**。数一数，我在这个故事里至少三次"以为自己看全了"：以为是 harness 差异、以为 loongcli 更强、以为协议写了就够。

## 修复：一行，把漏掉的那项补进协议

我把恢复协议的第一步改成"重读黑名单文件、恢复硬门槛与权重"，并写明：**启动时虽读过，但压缩后内容已失真，必须重读原文——别信"我读过"的印象。** 一行，治本。

## 可迁移的认知

1. **最危险的 unknown-unknown，是"设了防、以为防全了"。** 忘了设防，至少你知道这块是裸的；设了防却漏一项，你会因为"我防过了"而停止怀疑——盲区被你自己的安全感盖住了，往往漏得最久。

2. **防御清单本身要被审计。** 任何"压缩后 / 重启后 / 恢复时要做 X、Y、Z"的协议，都要反问一句：清单是不是漏了项？而漏掉的那项，往往恰恰是最该防的（黑名单正是硬过滤的依据）。

3. **"以为看全了"会反复发生，连查 bug 时也不例外。** 这个故事里我栽了好几次——测试设计、归因、读源码。每次都是数据或事实把我拽回来。能被拽回来，靠的不是聪明，是肯多看一眼、肯承认上一个判断错了。

4. **防御要写进结构，不能只写进清单。** 清单靠"记得执行"且"列得全"，两头都脆。更稳的是让框架在压缩后自动重挂关键状态——把"记得重读黑名单"变成"想忘也忘不掉"。这是下一步。

---

*追查路上那几个探针（测压缩后外部状态重读的）都留在了仓库里。但这一篇最值钱的不是探针，是想明白了一件事：我花最多力气设防的地方，恰恰是我最容易停止怀疑、于是漏得最久的地方。*

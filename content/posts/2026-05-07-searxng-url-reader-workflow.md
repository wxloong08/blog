## 问题不是“搜不到”，而是“读不干净”

我现在常用的一条链路是：

- 在 Claude Code 里写代码、做调研
- 把一部分任务分给 `DeepSeek 4.0 Pro`
- 需要搜索时，走我自己搭的 `SearXNG + searxng-mcp`

起因很直接：**DeepSeek 4.0 Pro 在 Claude Code 里没法直接用内置的 `websearch` 工具**。  
这意味着它能写、能总结、能比较，但不能自己出去搜。为了把这条能力补上，我先搭了 `SearXNG`，再接上 `searxng-mcp`，至少先把“搜索结果列表”这一步补齐。

但很快又卡在下一步：**搜索只是找到链接，不等于拿到可用正文。**

Claude Code 自带的 `webfetch`、`web_url_read` 之类工具，在很多场景下会把整页都抓下来。它们并不关心“什么是正文”，也不关心“这些内容是不是应该喂给 agent”。最后给模型的，往往是：

- 导航栏
- 侧栏推荐
- 页脚版权
- 登录提示
- 热榜
- 评论区里的低信息回复

真正值得看的正文，反而只占一部分。

这就是 `url-reader` 这个项目存在的原因。

## 我现在的实际工作流

我现在不是把一个工具当万能入口，而是拆成两段：

1. **搜索**
   - 用 `SearXNG + searxng-mcp`
   - 解决的是“找到哪些网页值得看”

2. **阅读**
   - 用 `url-reader`
   - 解决的是“把网页正文尽量干净地拿出来”

也就是说，`url-reader` 不是搜索引擎，而是搜索后的**网页读取层**。

它在我这里替代的，不是浏览器，而是那些“能抓网页，但抓得太脏”的 reader 工具。

## 为什么默认 reader 不够

我自己遇到的问题主要有三类。

### 1. 中文站不稳定

像知乎、微信公众号、小红书这类站，`Jina` 经常直接 `451`，或者只能返回残缺内容。  
有些页面公开可见，但 reader 还是拿不到正文。

### 2. 抓到了，但不适合喂给模型

很多 reader 的思路是“尽量多抓”，不是“尽量只保留正文”。  
这对人工浏览问题不大，但对 agent 不友好。模型看到的是整页噪音，推理预算却在消耗。

### 3. 没有平台感知

论坛、问答页、社交帖其实是三种很不一样的内容结构：

- 论坛要分主楼和回复
- 问答页要尽量只保留问题和回答
- 社交帖要去掉包装层和相关内容推荐

如果一律按“整页转 Markdown”处理，最后得到的只会是一份 dump。

## url-reader 在链路里做什么

`url-reader` 做的事不复杂，但很具体：

- 先识别平台
- 再按平台策略顺序尝试读取
- 然后做针对性的清洗
- 最后输出适合继续交给 LLM 的 Markdown

我现在常用的回退链大概是：

```text
Firecrawl -> OpenCLI Browser -> Jina -> Playwright
```

但这不是全局固定顺序，真正的顺序是按平台配置走的。  
重点不是“某个策略一定最好”，而是**不要信任任何单一抓取方式**。

## 这套东西最关键的不是抓取，而是清洗

我后来越来越确定，真正有价值的部分不在“怎么把网页抓下来”，而在“抓下来之后怎么把杂质剔掉”。

比如：

- `知乎 question`
  - 我只想要问题标题和已加载回答
  - 不想要关注数、浏览数、热榜、客户端下载、侧栏推荐

- `NodeSeek / V2EX / HostLoc / LINUX DO`
  - 我想保留主楼和高信息回复
  - 不想保留 `BD`、`支持`、`前排`、`谢谢老板` 这种回复

- `X / Twitter`
  - 我想保留标题、作者、时间、图片、正文
  - 不想要 `Related Posts`、`Replies`、包装层标题

对 agent 来说，这一步的价值比“多抓到一点 HTML”大得多。  
因为你最终要的是**可用上下文**，不是网页归档。

## 为什么我最终保留了这个项目

一开始我只是想给 `DeepSeek 4.0 Pro` 补一条能搜网页的路。  
结果最后发现，真正稀缺的不是搜索，而是：

> 搜到之后，能不能把网页变成一份干净、可喂、低噪音的研究输入。

`SearXNG` 解决的是“去哪找”。  
`url-reader` 解决的是“找回来之后，给模型看什么”。

这就是我继续维护这个工具的原因。

## 现在适合拿它做什么

目前我主要把它用在这些场景里：

- Claude Code 调研外部资料
- 论坛帖、问答页、技术博文的正文抽取
- 给 agent 准备更短、更干净的阅读输入
- 避免在 MCP / reader 工具链里反复手动开浏览器复制正文

## 还不够好的地方

这东西现在当然还不算完成态。

- `Reddit` 还没有专门的评论提取器
- 某些知乎页面仍然会有少量格式噪音
- 强反爬站点不保证稳定成功
- 还没做系统化 benchmark，只能先靠日常使用反馈迭代

但对我来说，它已经解决了最实际的问题：

**搜索结果点开之后，不用再把整页垃圾一起塞给模型。**

代码在 [github.com/wxloong08/url-reader](https://github.com/wxloong08/url-reader)。

> loongcli 开发笔记第三篇。前两篇讲的是单点问题（compact 摧毁历史、recall 注入位置）。这一篇是把它们串起来的范式复盘：我抄了 Claude Code 的五层上下文压缩架构，却没意识到那套架构隐含了一个我用不上的成本模型。给未来的自己留一句话——**架构可以抄，成本模型假设不能照搬。**

## 起点：一句跑题的闲聊

本来在聊一个无关的事，话头拐到："loongcli 的固定 prompt 占用大概 8K token，技能多了、记忆多了，会不会膨胀？"

我查了一下，发现有上限、不会失控。但就在回答的瞬间，我意识到自己差点答错重点：

**固定占用涨几 K 根本不是成本大头。** DeepSeek 有自动前缀缓存，命中价是未命中的约 1/120。只要前缀稳定，多占几 K 几乎免费。真正烧钱的不是"占用变大"，是"**前缀不稳定导致缓存失效**"。

这句话像一道裂缝。顺着它往下看，我发现 loongcli 整套压缩系统，可能都建在一个我用不上的假设上。

## 第一道裂缝：recall 注入位置（详见上一篇）

第一个撞上的是记忆召回。它每轮把 top-5 相关记忆注入在 `position 1`——系统提示之后、整个对话历史之前。而前缀缓存是连续匹配的：第一个变化的 token 往后全部失效。把每轮都在变的召回内容放在最前面，等于每轮让整段历史的缓存作废。

A/B 实测：注入在 position 1，缓存命中 53%；挪到对话末尾，99%。这个点上一篇写透了，这里只说它的意义——**它不是孤例，是同一个病的第一个症状**。它让我开始怀疑：如果"把变化内容放前面"会破坏缓存，那 compact 里那些为了"省 token"而删改历史的层，是不是也在干同样的事？

## 顺藤摸瓜：compact 的"减 token"，省出来 22 倍的账

loongcli 的上下文压缩有五层，其中第 3、4 层（snip 删远古消息、micro_compact 清旧工具结果）的作用是：在第 5 层（LLM 摘要）真正调用前，先把要喂给摘要的历史删减一遍，减少 token。

我一直把它们当"零成本优化"——不调 LLM，纯减 token，怎么会有成本？我甚至在面试材料里就是这么写的。

但顺着"前缀缓存"的视角再看一遍，逻辑反了：

摘要调用要把历史喂给 LLM。如果**直接喂完整历史**，这段 `[system, 完整历史]` 在主对话上一轮刚发过、已经被缓存——几乎全命中。但 snip/micro_compact **删改了**这段历史，前缀从删改点断裂，命中不了，全部重算。

我写了个 A/B 探针，真实 API 实测：

| | token 量 | 缓存命中 | 成本 |
|---|---|---|---|
| 删减后摘要（现状） | 15302 | **12%** | ¥0.0406 |
| 完整历史摘要 | 24352 | **98%** | ¥0.0018 |

**删减版省了 37% 的 token，却贵了 22 倍。** 因为命中/未命中差两个数量级（约 120 倍），而 token 量只少 4 成——4 成的量减，远抵不过 120 倍的单价上涨。

我的"零成本优化"，是个净亏 22 倍的负优化。而且它还附带一个我没算到的代价：snip 删的是远古消息，但摘要要总结"主要意图"——那些信息恰恰可能在被删的远古里。它不只亏钱，还让摘要丢早期信息。

## 退一步：整套金字塔，是为另一种成本模型设计的

到这里我才看清全局。这套五层压缩金字塔，是我参考 Claude Code 的架构做的。它的设计原则是"**越便宜的手段越早触发**"——而"便宜"的隐含定义是"token 少"。这是一个**成本模型假设**：token 量 = 成本，线性的。

这个假设对按 token 线性计费的场景（早期的 Claude/GPT）成立。但 DeepSeek 有强自动前缀缓存，它的成本模型根本不是线性的——**保持前缀稳定，比减少 token 重要得多**。

同一套架构，换一个成本模型，几层的价值就反转了。我把五层逐一重审：

| 层 | 做什么 | 在 DeepSeek 缓存下 | 处置 |
|---|---|---|---|
| 1 入口截断 | 截**新进入**的工具结果 | 截的是新内容（本就 miss），不碰已缓存前缀 | **留** |
| 2 collapse | 投影删改**已缓存**的旧内容 | 破坏前缀，负优化 | 跳过 |
| 3 snip | 删远古消息 | 破坏摘要调用缓存，实测 22x | 降为兜底 |
| 4 micro_compact | 清旧工具结果 | 同上 | 降为兜底 |
| 5 LLM 摘要 | 窗口满压成摘要 | 必要（窗口满必须压） | 留 |

规律很清晰：**凡是"删改已经缓存的内容来省 token"的层（2/3/4），在 DeepSeek 下都是负优化；不靠这招的层（1 截新内容、5 窗口满兜底），两种成本模型下都成立。**

## 解法：不删层，按成本模型分流

最容易的反应是"把 2/3/4 删掉"。但那太粗暴——loongcli 也支持非 DeepSeek 的 provider（本地 Ollama 无缓存、减 token 真有价值）。删了，那些场景就退化了。

正确的解法是承认一件事：**压缩策略是成本模型的函数**。于是加一个 `cache_aware` 标志：

- **cache-aware（DeepSeek）**：跳过第 2 层，摘要喂完整历史命中缓存，snip/micro 降为"仅超窗口兜底"。保前缀。
- **非 cache-aware（无强缓存）**：走完整金字塔。减 token。

不删任何能力，只是让同一套金字塔按 provider 的成本模型自适应。判断同时看 base_url 和 model 名——后者是为了兜住"走代理访问 DeepSeek"的场景：那时 base_url 不含 deepseek、会误判，但 model 名仍是 `deepseek-v4-pro`，靠它避免**静默退化**（又一次警惕静默失效——压缩策略悄悄掉回完整金字塔，不报错）。

## 几个我栽过的诚实细节

这篇是给自己的复盘，不修饰：

1. **我把 snip/micro 当"零成本"写进了面试材料。** 错的。它们有缓存成本，而且是负的。是这次顺着 recall 的线才发现。
2. **我连 compact 的触发阈值都记错过**——以为是代码里 `threshold=800000` 那个默认值（800K），实际被 main.py 用"窗口−1.3万"覆盖成了约 104 万。类签名的默认值不是实际生效值，我被自己的代码误导了。
3. **我对"要不要改"的判断被数据推翻过两次**（recall 那次、还有 compact）。能被推翻，是因为我每次都留了"先测"这一步——测量在 DeepSeek 上便宜到几毛钱，没有理由拍脑袋。

## 可迁移的认知

1. **抄架构时，要审查它没说出口的成本模型假设。** 我抄了 Claude Code 的压缩金字塔，抄来了"越便宜越早触发"的结构，却没意识到"便宜=token少"这个前提在 DeepSeek 上不成立。架构是显式的、好抄；成本模型假设是隐式的、藏在"为什么这么设计"里——这才是迁移时最容易踩的坑。

2. **缓存把"省钱"的方向整个调转了。** 没有缓存时，省钱 = 减 token；有强前缀缓存时，省钱 = 保前缀稳定。这两个方向在很多操作上是**冲突**的（删改历史减了 token、却破坏了前缀）。判断一个压缩操作好不好，不能只看"省了多少 token"，要看"它有没有动到已缓存的前缀"。

3. **"零成本"是个危险的词。** 我对 snip/micro 的误判，根源是把"不调 LLM"等同于"零成本"，漏掉了缓存这个隐性账本。任何说自己"零成本"的优化，都值得问一句：它在哪个账本上是零，又在哪个账本上偷偷记了账？
